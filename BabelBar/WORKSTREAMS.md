# BabelBar — разделение работы на два потока (агента)

Модель: **бесплатно + добровольная поддержка** (см. `MONETIZATION.md`). Никакого DRM, лицензий,
триала и серверного прокси. Два независимых потока.

## Репозитории / структура
- `BabelBar/` — **приложение** (Swift, текущий репо; внутреннее имя проекта остаётся
  `BabelBar`, отображаемое — **BabelBar**). Поток B.
- `web/` (новый, отдельный репозиторий) — **лендинг** (Next.js/Vercel, классический CSS). Поток A.
- Общие договорённости (ссылки/ассеты) — в конце этого файла.

---

## Поток A — Landing (агент 1)

**Зона ответственности:**
- Лендинг по `LANDING_BRIEF.md`: **Next.js на Vercel, классический CSS (CSS Modules + переменные),
  БЕЗ Tailwind**. Дружеский support-тон.
- **Только реальные скрины/записи приложения** (из `web/public/screens/`, предоставит заказчик) —
  никаких сгенерированных иллюстраций (исключение — маскоты в блоке Community/Support).
- **Поддержка:** кнопки-уровни (Coffee/Support/Supporter) ведут на страницу донатов
  (**Buy Me a Coffee** / **Boosty** / Lemon Squeezy — выбрать). Env: `SUPPORT_URL_*`.
- **Download:** ссылка на нотаризованный `.dmg` из GitHub Releases. Env: `DMG_URL`.
- Домен: отдельный Vercel-проект; целевой **`babelbar.app`** (до покупки — `*.vercel.app`).
- SEO, OG, аналитика (Plausible/Vercel), адаптив, a11y, perf (Lighthouse 90+).

**Не трогает:** Swift-код приложения.

**Definition of Done:** `LANDING_BRIEF.md` §9 — все секции собраны, Download ведёт на `.dmg`,
кнопки поддержки открывают платёжную страницу, деплой на Vercel с HTTPS.

---

## Поток B — Product (агент 2)

**Зона ответственности:**
- **Убрать триал/лицензионную заглушку:** секцию «LICENSE» в настройках заменить на мягкую
  **«♥ Support BabelBar»** — кнопка открывает `SUPPORT_URL` в браузере. Удалить из `AppState`
  логику триала/ключа (`trialDaysLeft`, `licenseKey`, `activateLicense` и т.п.).
- **Дистрибуция:** в `build.command` добавить подпись **Developer ID** + **notarization** + staple;
  собрать **`.dmg`** для GitHub Releases.
- **Автообновление (опц.):** Sparkle + appcast.
- Полировка продукта и баги. Внутренние имена (target/scheme/bundle id) НЕ переименовываем —
  только отображаемое имя BabelBar (уже сделано).

**Не трогает:** лендинг.

**Definition of Done:** в настройках — кнопка Support вместо LICENSE; нотаризованный `.dmg`
запускается без警告 Gatekeeper; весь функционал доступен без ограничений.

## Общие договорённости (между A и B)

- **`SUPPORT_URL`** — единая страница поддержки (например, Buy Me a Coffee/Boosty). Используется
  и на лендинге, и в приложении (кнопка Support). Зафиксировать после выбора платформы.
- **`DMG_URL`** — ссылка на последний релиз `.dmg` (GitHub Releases). Лендинг берёт её для Download.
- **Скрины** — заказчик кладёт реальные PNG/записи в `web/public/screens/` для потока A.
- Никаких секретов/ключей/серверных контрактов в этой модели не требуется.

## Порядок запуска
1. Выбрать платформу донатов → зафиксировать `SUPPORT_URL` (+ уровни).
2. Поток B: убрать триал, добавить Support-кнопку, настроить notarization, собрать `.dmg` → Release.
3. Поток A: собрать лендинг, вставить реальные скрины, подключить `SUPPORT_URL` и `DMG_URL`, задеплоить.
4. Купить `babelbar.app` → сделать primary-доменом проекта.

## Анти-скоуп
- Никакого DRM, лицензий, триал-гейтинга, серверного прокси и учёта токенов на нашей стороне.
- Не переименовывать внутренние идентификаторы проекта/бандла.
